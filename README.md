# FraudLens — Fine-grain fraud detection demo

This repository contains a full-stack student project for "Detecting fine grain fraud cases in large financial dataset using deep learning models." It provides a FastAPI backend (data ingest, lightweight dataset storage, and scoring job stubs) and a React + TypeScript frontend (dashboard, upload UI, and investigation panel).

This README contains quickstart steps, project structure, API documentation, data format expectations, testing notes, and next steps you can follow to continue development.

**Status:** Prototype / school project — includes working CSV/XLSX ingestion, dataset persistence to SQLite, a frontend UI shell, and CI + basic tests. Model training and production-grade scoring are planned next steps.

**Project overview**

FraudLens (Fine-Guard AI) is a small end-to-end prototype that demonstrates how to ingest transactional datasets (CSV/XLSX), persist them to a local SQLite store, run background scoring jobs (currently stubbed), and visualize flagged transactions on a React dashboard. The repository is intended as a student project scaffold for iterating into a full deep-learning based fraud detection pipeline.

Key goals:

- Support batch CSV/XLSX ingestion with automatic column normalization
- Persist datasets and transactions into SQLite for downstream processing
- Provide a clean, professional UI for analysts to inspect flagged transactions
- Make it easy to extend with a PyTorch training pipeline and real model serving

**Quickstart - Backend**

Requirements: Python 3.11+ (project uses 3.12 in CI) and the venv workflow.

From the repository root:

```powershell
python -m venv .venv
. .venv\Scripts\Activate.ps1
pip install -r requirements.txt
pip install -r requirements-dev.txt

# Run the FastAPI app (reload helpful during development)
cd backend
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

The API will be available at `http://localhost:8000`. Open `http://localhost:8000/docs` for FastAPI autogenerated OpenAPI UI (useful for testing endpoints directly).

**Quickstart - Frontend**

Requires Node.js (16+) and npm.

```powershell
cd frontend
npm install
npm run dev

# Open the dev server (usually http://localhost:5173)
```

The frontend is already wired to the backend dataset upload endpoint; by default it hits `http://localhost:8000/api/v1/datasets/`. You can change the API base in `frontend/.env` or `VITE_API_BASE` environment variable.

**API reference (important endpoints)**

- `GET /health` — basic health check
- `GET /ready` — readiness check (verifies DB connectivity)
- `POST /api/v1/datasets/` — upload a dataset file (CSV or XLSX). Returns `{ dataset_id, filename, row_count, status, warnings }`.
- `POST /api/v1/datasets/{dataset_id}/score` — queue a scoring job for a dataset (currently runs a stub background job)

Note: Use the OpenAPI docs at `/docs` to explore schemas and additional endpoints implemented under `/api/v1`.

**Data format and upload rules**

- Supported file types: `.csv`, `.xlsx`, `.xls`.
- Max rows: 200,000 (configurable in backend constant `MAX_ROWS`).
- Normalization: the backend attempts to map common column names into a standard schema: `occurred_at` (timestamp), `amount`, `merchant`, `channel`. If timestamps are missing or unparsable, current UTC timestamps are used. Amounts are coerced to numeric values (non-parsable values default to `0.0`).
- Uploaded transactions are stored in the local SQLite DB under `data/fineguard.db` by default.

**Development notes**

- Virtualenv: use the venv steps above to create an isolated Python environment.
- Tests: run `pytest -q` from the repo root. A basic health endpoint test is included.
- Linting / formatting: not yet enforced in CI but adding `ruff`/`black` or `prettier` is recommended.
- CI: GitHub Actions workflow added at `.github/workflows/ci.yml` — it installs dependencies and runs pytest.

**Project structure (top-level)**

- `backend/app/` — FastAPI application package
  - `api/` — routing and endpoints (v1 router and system endpoints)
  - `core/` — config and settings
  - `db/` — async DB session and base models
  - `models/` — SQLAlchemy models (Dataset, Transaction, Job)
  - `schemas/` — Pydantic response/request models
  - `services/` — background jobs and scoring stubs
- `frontend/` — React + TypeScript UI (Vite)
- `data/` — default location for the SQLite DB (data/fineguard.db)
- `tests/` — pytest test files
- `.github/workflows/ci.yml` — CI pipeline

**Model: training and usage**

This project now includes a minimal demonstrator PyTorch model and training pipeline to help you iterate quickly. It is intentionally small so it runs on a laptop and is suitable for learning and extension.

- Training CLI: `backend/train.py` — generates synthetic transaction data and trains a demo MLP.

From the repository root, run the trainer from the `backend` folder:

```powershell
cd backend
python train.py --rows 5000 --out data/synth.csv --model-name demo.pt --epochs 6
```

This will:
- generate `data/synth.csv` (synthetic transactions with a `label` column)
- train a small MLP and save model weights to `backend/models/demo.pt` and metadata to `backend/models/demo.pt.meta`

Prediction (API): the backend exposes a prediction preview endpoint which applies the trained model to transactions from a dataset stored in the DB.

- `POST /api/v1/predict/{dataset_id}` — returns a small preview of model scores for the dataset (requires a trained model to exist in `backend/models`). If the model is missing, the endpoint returns HTTP 500 with an explanatory message.

Programmatic inference: you can also use the model service directly from Python:

```python
from app.services.model import predict_df, load_model
import pandas as pd

df = pd.read_csv('some_transactions.csv')
scores = predict_df(df, save_name='demo.pt')
```

Files added for model/demo:
- `backend/app/services/synth.py` — synthetic transaction generator
- `backend/app/services/model.py` — PyTorch model, train/save/load/predict helpers
- `backend/train.py` — small CLI that ties generator + training together
- `backend/app/api/v1/endpoints/predict.py` — API endpoint for prediction previews

Notes and caveats:
- This model is a learning scaffold, not production-ready. Feature engineering is minimal (amount + categorical encodings) and model capacity is small.
- For real tasks, add proper feature pipelines, class imbalance handling, evaluation metrics, model versioning, and saving of preprocessing artifacts.
- If you plan to train on large datasets, move training off-device or implement streaming/batched training.
